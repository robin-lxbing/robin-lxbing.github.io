Electron解析

发表于2026-04-01|更新于2026-04-15|前端开发
|总字数:7.5k|阅读时长:22分钟|浏览量:

很多开发者对 Electron 的认知停留在“HTML+CSS+JS 写桌面应用”的表层，却忽略了其底层架构设计的深层逻辑：为什么要拆分主进程与渲染进程？IPC 通信为何不能像前端跨页面通信那样简单？Chromium 与 Node.js 的组合到底带来了哪些不可替代的优势，又埋下了哪些性能隐患？

一、核心原理：不是“前端套壳”，是 Chromium 与 Node.js 的深度融合

很多人吐槽 Electron 是“前端套壳”，本质上是对其核心设计的误解。Electron 的核心价值，从来不是“用前端技术写桌面应用”，而是将成熟的 Web 生态与原生桌面能力做了无感知的桥接——它没有重新造轮子，而是将 Chromium（负责 UI 渲染与 Web 标准兼容）和 Node.js（负责原生系统交互与底层能力）两个成熟的 runtime 深度整合，再通过一套自定义的通信机制与生命周期管理，形成了一套完整的跨平台桌面应用开发体系。

这种设计的底层逻辑，源于跨平台开发的核心痛点：传统桌面应用（C++/Qt/Swift）需要为不同系统（Windows/macOS/Linux）编写三套代码，开发成本高、迭代效率低；而 Web 应用虽能跨平台，却受限于浏览器沙箱，无法访问本地文件、系统托盘、原生菜单等桌面能力。Electron 的解决方案，是“扬长避短”：复用前端开发者熟悉的技术栈（HTML/CSS/JS）降低学习成本，借助 Chromium 实现跨平台 UI 一致性，依托 Node.js 突破浏览器沙箱限制，获得原生系统交互能力。

这里有一个关键细节：Electron 并非简单地将 Chromium 和 Node.js 打包在一起，而是对两者的运行时做了深度改造。最典型的就是 V8 引擎的复用——Chromium 和 Node.js 都基于 V8 执行 JavaScript，但 Electron 让两者共享同一个 V8 实例（不同进程间通过 IPC 同步状态），避免了重复加载 V8 带来的内存浪费。这也是为什么 Electron 能实现“前端代码直接调用 Node.js 模块”，而不是通过第三方接口间接通信的核心原因。

举个真实案例：我们曾开发一款面向设计师的素材管理工具，需要同时实现“Web 端预览素材”（依赖 Chromium 的 WebGL 渲染）和“本地文件批量处理”（依赖 Node.js 的 fs 模块与流处理）。如果用传统方案，要么用 Qt 写原生界面（前端开发者上手成本高），要么用 Web 应用+本地服务（跨平台部署复杂，且存在跨域与权限问题）。而 Electron 让我们直接在渲染进程中实现预览逻辑，在主进程中处理本地文件操作，通过 IPC 传递数据，既复用了前端团队的技术积累，又实现了原生级的系统交互，开发周期缩短了近 40%。

二、进程模型：Main/Renderer 拆分的底层逻辑与工程实践

Electron 继承了 Chromium 的多进程架构，核心拆分为主进程（Main Process）和渲染进程（Renderer Process），这种拆分不是随意设计，而是为了平衡“稳定性、安全性与性能”——这一点，从 Chrome 浏览器的多标签页进程模型就能找到根源：早期浏览器采用单进程模型，一个标签页崩溃会导致整个浏览器挂掉，而多进程模型能实现“故障隔离”，单个页面崩溃不影响全局。

Electron 沿用这一思路，并结合桌面应用的特性做了调整，明确了两个进程的核心职责与边界，而这种边界的划分，本质上是“原生能力与 Web 能力的隔离”。

主进程（Main Process）是应用的入口，运行在 Node.js 环境中，拥有完整的系统权限——负责应用的生命周期管理（启动、退出、窗口创建/销毁）、原生资源调用（文件系统、系统托盘、菜单、对话框）、全局状态管理，以及与操作系统的底层交互。每个 Electron 应用只有一个主进程，它是整个应用的“中枢”，所有涉及原生能力的操作，最终都必须通过主进程完成。

为什么主进程不能直接渲染 UI？核心原因是“稳定性”——主进程如果负责渲染，一旦 UI 渲染出现异常（如 JS 死循环、DOM 操作错误），会导致整个应用崩溃；而将渲染逻辑拆分到独立的渲染进程，即使某个窗口的渲染进程崩溃，主进程依然能正常运行，用户可以关闭异常窗口，避免应用整体挂掉。这一点在大型应用中尤为重要，比如 VS Code（基于 Electron 开发），即使某个编辑器窗口崩溃，主进程依然能保留其他窗口的状态，不会导致用户代码丢失。

渲染进程（Renderer Process）对应每个打开的窗口，本质上是一个轻量级的 Chromium 渲染进程，运行在浏览器环境中，负责 UI 渲染、前端交互逻辑（事件绑定、DOM 操作、前端框架渲染）。默认情况下，渲染进程没有 Node.js 权限，也无法直接访问原生资源——这种设计是为了安全，避免恶意前端代码（如第三方插件、注入脚本）直接操作本地系统，引发安全风险。

这里有一个常见的工程实践误区：很多开发者为了图方便，通过设置 nodeIntegration: true 让渲染进程直接访问 Node.js 模块，这种做法在小型demo中可行，但在生产环境中极其危险。我们曾接手一个遗留项目，前序团队开启了 nodeIntegration，导致第三方插件注入的恶意脚本通过 fs 模块窃取用户本地文件，最终不得不重构进程通信逻辑，关闭该配置，增加了大量开发成本。

此外，Electron 还会根据需求启动其他辅助进程，比如 GPU 进程（负责图形渲染加速，优化 UI 动画与 3D 效果）、插件进程（处理第三方插件）、沙箱进程（隔离危险操作），这些进程由主进程统一管理，进一步提升了应用的稳定性与安全性。

三、IPC 通信：进程隔离下的“数据桥梁”，为什么不能简单复用前端通信方案？

主进程与渲染进程的隔离，必然带来一个核心问题：如何实现数据传递与指令交互？这就是 IPC（Inter-Process Communication，进程间通信）的核心价值。很多前端开发者会疑惑：为什么不能用 localStorage、sessionStorage 或者 WebSocket 实现通信？答案很简单：这些方案要么受限于进程隔离（localStorage 是进程内存储，无法跨进程共享），要么存在性能开销（WebSocket 基于网络协议，延迟高于原生 IPC），无法满足桌面应用的交互需求。

Electron 提供了两套核心 IPC 通信 API：主进程的 ipcMain 和渲染进程的 ipcRenderer，支持同步与异步通信，其底层基于 Chromium 的 IPC 机制实现，本质上是“消息队列+内存共享”的组合——同步通信采用“阻塞式消息传递”，适用于简单的、需要立即获取结果的场景（如获取应用配置）；异步通信采用“非阻塞式消息传递”，适用于耗时操作（如批量处理本地文件），避免阻塞 UI 渲染。

设计 IPC 机制时，Electron 重点解决了三个核心问题：安全性、性能与易用性。安全性上，通过“信道（Channel）”划分通信范围，开发者可以自定义信道名称，避免消息泄露；性能上，采用二进制序列化（默认使用 JSON 序列化，可自定义 Protocol Buffers 提升效率），减少数据传输开销；易用性上，API 设计贴合前端开发者习惯，无需关注底层进程通信细节，只需调用 send、on 等方法即可实现通信。

结合我们的项目实践：在一款办公协同工具中，我们需要实现“渲染进程触发本地文件上传，主进程处理文件校验与上传，渲染进程实时显示进度”的功能。这里采用异步 IPC 通信，渲染进程通过 ipcRenderer.send 向主进程发送上传指令与文件路径，主进程通过 ipcMain.on 接收指令，处理文件校验、分片上传，再通过 webContents.send 向渲染进程发送进度信息，渲染进程更新 UI 显示。整个过程中，异步通信避免了 UI 阻塞，而信道的隔离的设计，确保了上传指令不会被其他窗口的渲染进程误接收。

需要注意的是，IPC 通信并非无代价：频繁的 IPC 调用会增加进程间的上下文切换开销，尤其是传递大量二进制数据（如图片、视频）时，容易导致 UI 卡顿。我们曾遇到过一个问题：渲染进程频繁向主进程发送图片数据，导致 IPC 队列阻塞，渲染进程帧率从 60fps 下降到 20fps 以下。最终的解决方案是：将图片数据先在渲染进程中进行压缩处理，再批量发送，同时优化 IPC 调用频率，减少不必要的消息传递——这也印证了：IPC 是“桥梁”，但过度依赖会成为性能瓶颈。

另外，Electron 早期提供的 remote模块，本质上是 IPC 的“封装简化版”，允许渲染进程直接调用主进程的 API（如创建窗口、调用原生模块）。但这种设计存在严重的安全风险：渲染进程若被注入恶意代码，可通过 remote 模块直接操控主进程，获取系统权限；同时，remote 模块的底层实现会创建大量的跨进程引用，容易导致内存泄漏。因此，Electron 从 v14 版本开始，将 remote 模块标记为废弃，推荐使用contextBridge + 预加载脚本（preload.js）的方式，实现渲染进程与主进程的安全通信。

四、性能开销：Chromium + V8 + Node 组合的“甜蜜负担”

Electron 最大的争议，莫过于其性能与资源占用——“启动慢、内存高、安装包大”，几乎是所有 Electron 应用的共性问题。但很少有开发者深入思考：这些性能开销到底来自哪里？是框架设计的缺陷，还是工程实践的不足？

本质上，Electron 的性能开销，源于“Chromium + V8 + Node.js”的组合架构——这三个组件各自都是重量级选手，叠加后必然带来资源消耗的叠加，而这种叠加，是为了实现“跨平台+前端复用+原生能力”所付出的必要代价。

首先是安装包体积。Electron 应用的安装包，必须包含 Chromium 渲染引擎、Node.js 运行时和应用代码，其中 Chromium 占比最大（约 100MB 以上），Node.js 占比约 20-30MB，即使是一个最简单的“Hello World”应用，打包后体积也会超过 120MB（Windows 平台）。这也是为什么很多轻量级工具（如简单的文本编辑器）不适合用 Electron 开发——安装包体积过大，会显著提升用户的下载成本。我们曾开发一款轻量级日志查看工具，用 Electron 打包后体积达 130MB，而用 Qt 开发的同类工具，体积仅 10MB 左右，最终不得不切换技术栈。

其次是内存占用。Electron 应用的内存占用，主要来自三个方面：Chromium 渲染进程的内存消耗（每个窗口对应一个渲染进程，即使是空白窗口，也会占用约 50-100MB 内存）、Node.js 运行时的内存消耗（约 20-50MB）、以及进程间通信与状态同步带来的内存开销。以 VS Code 为例，打开 5-6 个标签页，内存占用通常会达到 500MB 以上，这也是很多低配设备用户吐槽 VS Code 卡顿的核心原因。

这里需要澄清一个误区：内存高不等于“性能差”。Electron 的内存占用高，很大程度上是“预加载与缓存”的结果——Chromium 会预加载常用的渲染资源，Node.js 会缓存模块，目的是提升后续操作的响应速度。比如，第一次打开 Electron 应用启动慢，但第二次启动会快很多，就是因为 Chromium 和 Node.js 的缓存生效了。但这种“以内存换速度”的策略，在低配设备上会适得其反，导致系统卡顿。

再来看启动速度。Electron 应用的启动过程，本质上是“启动 Node.js 主进程 → 初始化 Chromium 引擎 → 创建渲染进程 → 加载应用代码 → 渲染 UI”的过程，每个步骤都有一定的耗时。其中，Chromium 引擎的初始化是最耗时的环节——尤其是在低配设备上，可能需要 3-5 秒甚至更久。我们曾优化一款 Electron 应用的启动速度，从最初的 4.5 秒优化到 1.8 秒，核心优化点就是：减少主进程初始化时的模块加载（只加载必要模块）、禁用 Chromium 的不必要功能（如 GPU 加速、插件支持）、预编译应用代码（减少启动时的解析耗时）。

此外，V8 引擎的垃圾回收（GC）也是性能开销的重要来源。Electron 中，主进程和渲染进程各自拥有独立的 V8 实例，GC 操作会占用 CPU 资源，若应用中存在内存泄漏（如未销毁的 BrowserWindow 实例、未解绑的 IPC 监听器），会导致 GC 频繁触发，CPU 占用率飙升，进而导致 UI 卡顿。我们曾遇到过一个严重的内存泄漏问题：在渲染进程中频繁创建定时器，且未在组件卸载时清除，导致渲染进程内存持续增长，GC 每 10 秒触发一次，CPU 占用率高达 80% 以上，最终通过排查定时器与 IPC 监听器，修复了泄漏问题。

总结来说，Electron 的性能开销，是“跨平台+前端复用”的必然代价——它不是框架设计的缺陷，而是取舍后的结果。对于复杂的生产力工具（如 VS Code、Figma），这种代价是可接受的，因为其带来的开发效率提升和跨平台一致性，远大于性能开销；但对于轻量级工具、低配设备适配的应用，这种代价就显得得不偿失。

五、内存管理：多进程模型下的“坑点”与工程优化策略

Electron 的内存管理，本质上是“多进程内存隔离与共享”的管理——主进程、渲染进程、辅助进程各自拥有独立的内存空间，同时通过 IPC 实现少量数据的共享，这种模型既带来了稳定性优势（单个进程内存泄漏不会影响全局），也增加了内存管理的复杂度。

首先要明确：Electron 的内存泄漏，大概率不是框架本身的问题，而是开发者的工程实践不当导致的。结合我们的项目经验，最常见的内存泄漏场景主要有三类：

第一类是主进程中的 BrowserWindow 实例泄漏。很多开发者在创建窗口后，未在窗口关闭时释放引用，导致 BrowserWindow 实例无法被 GC 回收。比如，未在 closed 事件中将窗口实例置为 null，即使窗口已经关闭，实例依然占用内存。我们曾接手一个项目，前序团队创建窗口后未释放引用，导致用户多次打开关闭窗口后，主进程内存占用突破 1GB，最终通过在 closed 事件中置空窗口实例，内存占用下降了 60%。

第二类是 IPC 监听器泄漏。使用 ipcMain.on 或 ipcRenderer.on 绑定监听器后，若未在适当的时候解绑（如窗口关闭、组件卸载），会导致监听器堆积，进而导致内存泄漏。比如，在渲染进程的组件中绑定 IPC 监听器，组件卸载时未调用 removeListener，会导致组件实例无法被回收。解决方案是：要么使用 ipcRenderer.once（只监听一次，自动解绑），要么在组件卸载时显式解绑监听器。

第三类是 Node.js 模块与原生模块的引用泄漏。Node.js 的模块缓存机制（require 后模块会被缓存），若频繁加载大型模块且未释放引用，会导致内存持续增长；而原生模块（C++ 插件）若未正确释放 V8 对象引用，会导致内存泄漏无法被 GC 检测到。比如，我们曾使用一个第三方原生模块处理图片，该模块未正确释放 V8 对象引用，导致应用运行一段时间后，内存占用持续飙升，最终通过替换模块并优化引用释放逻辑，解决了泄漏问题。

除了避免内存泄漏，Electron 内存管理的另一个核心是“内存优化”。结合工程实践，我们总结了三个有效的优化策略：一是“进程复用”，减少不必要的窗口创建（如使用标签页替代多窗口），降低渲染进程的内存消耗；二是“资源懒加载”，主进程初始化时只加载必要模块，渲染进程只加载当前窗口所需的资源，避免一次性加载所有资源；三是“主动 GC 触发”，在耗时操作完成后（如批量处理文件），通过 global.gc()（需启动时添加 --expose-gc 参数）强制执行 GC，释放内存。

需要注意的是，主动 GC 不能滥用——频繁触发 GC 会占用 CPU 资源，导致 UI 卡顿，需根据应用场景合理使用。比如，在文件处理完成后触发一次 GC，而不是在循环中频繁触发。

六、打包机制：跨平台分发的核心，从原理到工程实践

Electron 应用的打包，本质上是“将应用代码、Electron 运行时（Chromium + Node.js）、依赖模块打包成目标平台可执行文件”的过程——其核心价值是实现“一次开发，多端分发”，让开发者无需为不同平台单独适配打包逻辑。

Electron 的打包机制，依赖于第三方打包工具，其中最主流的是 electron-packager 和electron-builder。两者的核心区别的是：electron-packager 更轻量，专注于“打包”本身，将应用代码与 Electron 运行时捆绑，生成目标平台的可执行文件（如 .exe、.app），但不支持自动生成安装包（如 .msi、.dmg）；electron-builder 功能更全面，不仅支持打包，还能自动生成安装包、处理图标、签名、更新等功能，是生产环境中更常用的选择。

打包的底层逻辑的是：针对不同平台，Electron 会下载对应平台的 Electron 运行时（包含 Chromium 和 Node.js），然后将应用代码（主进程、渲染进程、预加载脚本等）复制到运行时的指定目录，再通过打包工具修改运行时的配置（如应用名称、图标、权限），最终生成可执行文件。这里的关键是“平台适配”——不同平台的可执行文件格式、权限要求、资源路径都不同，打包工具会自动处理这些差异，比如 Windows 平台生成 .exe 文件，macOS 平台生成 .app 文件，Linux 平台生成 .AppImage 文件。

工程实践中，打包最常见的问题有两个：一是打包体积过大，二是跨平台适配异常。针对打包体积过大，我们的优化策略是：通过 --ignore 参数过滤不必要的文件（如 .git、日志文件、开发环境依赖），使用 electron-builder 的压缩功能（如 UPX 压缩），拆分依赖模块（将大型第三方模块按需加载，而非打包进主程序）。比如，我们开发的办公协同工具，最初打包体积达 200MB，通过过滤无用文件、压缩主程序、拆分依赖，最终将体积优化到 120MB 左右。

针对跨平台适配异常，核心是“遵循平台规范”。比如，macOS 平台要求应用签名（否则无法在 macOS 10.15+ 上运行），需要在打包时配置签名证书；Windows 平台要求应用图标格式为 .ico，且需适配不同尺寸；Linux 平台需要处理权限问题（如文件读写权限、桌面快捷方式）。我们曾遇到过一个问题：macOS 平台打包后，应用无法启动，排查后发现是未配置签名证书，导致系统阻止应用运行，配置签名后问题解决。

此外，打包后的应用更新机制，也是工程实践中不可忽视的环节。Electron 本身不提供更新功能，但可以通过第三方模块（如 electron-updater）实现自动更新——其核心原理是：应用启动时，向服务器请求最新版本信息，若有更新，下载更新包，然后替换当前应用文件，重启应用完成更新。需要注意的是，跨平台更新的逻辑不同，比如 Windows 平台可以直接替换 .exe 文件，而 macOS 平台需要处理 .app 文件的签名与权限问题。

七、安全问题：沙箱、contextIsolation 与工程安全实践

Electron 的安全问题，本质上是“Web 环境与原生环境的边界安全”——渲染进程运行在浏览器环境中，若没有严格的隔离机制，恶意代码（如第三方插件、注入脚本）可能通过 Node.js 模块获取系统权限，引发安全风险。Electron 的安全设计，核心是“最小权限原则”，通过沙箱（sandbox）、contextIsolation 等机制，限制渲染进程的权限，隔离 Web 环境与原生环境。

首先是沙箱（sandbox）机制。沙箱是 Chromium 提供的一种安全机制，用于隔离渲染进程与系统资源——启用沙箱后，渲染进程将无法直接访问本地文件、系统 API，所有涉及原生资源的操作，必须通过主进程间接完成。Electron 中，沙箱机制默认是关闭的，需要通过配置 sandbox: true 启用。启用沙箱后，渲染进程的权限被严格限制，即使被注入恶意代码，也无法直接操作系统资源，从而提升应用的安全性。

但沙箱机制并非“万能”，启用后会带来一些功能限制——比如，渲染进程无法直接使用 Node.js 模块，无法访问本地文件，需要通过预加载脚本（preload.js）与主进程通信，间接获取所需资源。这也是为什么很多开发者不愿意启用沙箱的原因——开发成本增加，但从安全角度出发，生产环境中必须启用沙箱，尤其是涉及用户敏感数据（如密码、本地文件）的应用。

其次是 contextIsolation（上下文隔离）机制。contextIsolation 是 Electron 提供的另一种安全机制，用于隔离渲染进程的主上下文（main world）与预加载脚本的上下文（isolated world）。启用 contextIsolation 后，预加载脚本中定义的变量、函数，不会暴露到渲染进程的全局作用域（window）中，避免恶意代码通过全局作用域获取预加载脚本中的敏感信息（如 IPC 通信方法、Node.js 模块引用）。

contextIsolation 从 Electron v12 版本开始默认启用，其核心设计逻辑是：预加载脚本是“桥梁”，负责连接渲染进程与主进程，但不能成为恶意代码的“突破口”。比如，预加载脚本中通过 contextBridge 暴露通信方法，即使渲染进程被注入恶意代码，也无法直接访问 contextBridge 中的其他逻辑，只能调用暴露的方法，从而限制了恶意代码的权限。

这里结合我们的安全实践：在开发一款涉及用户密码管理的应用时，我们启用了沙箱与 contextIsolation，预加载脚本中通过 contextBridge 暴露仅用于密码验证的 IPC 方法，渲染进程只能调用该方法，无法直接访问 Node.js 的 crypto 模块（用于密码加密），也无法访问本地文件（用于密码存储）。这种设计，即使渲染进程被注入恶意代码，也无法获取用户密码，有效提升了应用的安全性。

除了沙箱与 contextIsolation，还有几个关键的安全实践点需要注意：一是禁用 nodeIntegration，避免渲染进程直接访问 Node.js 模块；二是禁用 enableRemoteModule，避免使用废弃的 remote 模块带来的安全风险；三是验证所有 IPC 消息，避免恶意消息注入（如验证消息来源、消息格式）；四是对用户输入进行过滤，避免 XSS 攻击（虽然渲染进程运行在 Chromium 中，有 XSS 防护，但仍需谨慎）。

需要强调的是，Electron 的安全问题，大多不是框架本身的漏洞，而是开发者的配置不当导致的。很多开发者为了图方便，禁用沙箱、启用 nodeIntegration，本质上是牺牲安全换取开发效率，这种做法在生产环境中极其危险——一旦应用被攻击，用户的敏感数据可能被窃取，应用的声誉也会受到严重影响。

八、适用边界：客观看待 Electron 的优势与代价

经过多年的工程实践，我始终认为：Electron 不是“万能的跨平台框架”，它有明确的适用边界，其价值取决于项目的需求——在合适的场景中，它能大幅提升开发效率；在不合适的场景中，它的性能与资源开销会成为致命缺陷。

Electron 的核心优势，在于“前端复用+跨平台一致性+原生能力”的平衡：

  1. 开发效率极高：前端开发者无需学习新的技术栈，即可快速开发跨平台桌面应用，复用前端生态的大量组件与工具（如 React、Vue、WebPack），大幅缩短开发周期。比如，我们团队以前端开发者为主，开发一款跨平台办公工具，仅用 3 个月就完成了 Windows、macOS、Linux 三个平台的开发与上线，若用传统原生技术，至少需要 6 个月以上。
  2. 跨平台一致性好：依托 Chromium 引擎，应用在不同平台上的 UI 渲染与交互体验高度一致，无需为不同平台单独适配 UI 逻辑，减少了跨平台适配的成本。这一点对于大型应用（如 VS Code、Figma）尤为重要，能确保全球用户获得一致的使用体验。
  3. 原生能力完善：通过 Node.js 与 Electron 提供的原生模块，能实现几乎所有桌面应用的功能（文件操作、系统托盘、菜单、对话框、热更新等），满足复杂生产力工具的需求。

而 Electron 的核心代价，在于“性能与资源占用”：

  1. 资源占用高：安装包体积大、内存占用高、启动速度慢，不适合轻量级工具、低配设备适配的应用，也不适合对资源占用敏感的场景（如嵌入式设备、老旧 PC）。
  2. 性能上限有限：受限于 Chromium 与 Node.js 的组合架构，即使经过优化，其性能也无法与原生桌面应用（C++/Qt）相比，不适合高性能需求的应用（如视频剪辑、3D 建模工具）。

结合场景来看，Electron 最适合的是“复杂生产力工具、办公协同工具、内容管理工具”——这类应用的核心需求是“跨平台、开发效率、功能完善”，对性能的要求相对不高，Electron 的优势能得到充分发挥。比如 VS Code（代码编辑器）、Figma（设计工具）、Slack（协同工具），都是 Electron 应用的成功案例。

而以下场景，不推荐使用 Electron：

  1. 轻量级工具（如简单的文本编辑器、计算器）：这类应用功能简单，用户对安装包体积和内存占用敏感，用更轻量的框架（如 Tauri、Qt）更合适。
  2. 高性能需求应用（如视频剪辑、3D 建模、游戏）：这类应用对 CPU、GPU 性能要求极高，Electron 的性能开销会导致体验不佳，应选择原生技术或更轻量的跨平台框架（如 Unreal Engine、Qt）。
  3. 嵌入式设备、低配设备适配的应用：这类设备资源有限，Electron 的高内存占用和启动速度慢，会导致应用无法正常运行。

此外，随着 Tauri 等轻量级跨平台框架的兴起，Electron 的市场份额正在受到冲击——Tauri 依托系统原生 WebView（而非自带 Chromium）和 Rust 原生后端，安装包体积和内存占用远低于 Electron，安全性也更高。但 Tauri 也有其局限性：学习成本高（需要掌握 Rust）、生态成熟度不如 Electron、对老旧系统的兼容性较差。因此，选型时需根据项目需求、团队技术栈综合判断，而非盲目跟风。
